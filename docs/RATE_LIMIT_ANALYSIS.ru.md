## Ключевые выводы

### 1. Шаблоны ошибок ограничения частоты запросов

Когда команды GitHub CLI (`gh`) достигают ограничений частоты запросов, они генерируют специфические сообщения об ошибках, которые можно обнаружить:

#### Основная ошибка ограничения частоты запросов

```
HTTP 403: You have exceeded a secondary rate limit. Please wait a few minutes before you try again. If you reach out to GitHub Support for help, please include the request ID D84A:2DE4CB:565BC8C:5079198:68CB676D. (https://api.github.com/search/issues?advanced_search=true&page=3&per_page=100&q=is%3Aissue+type%3Aissue)
```

#### Шаблоны обнаружения

Для обнаружения ошибок ограничения частоты запросов проверяйте следующие шаблоны в сообщениях об ошибках:

- `"rate limit"` (без учёта регистра)
- `"secondary rate limit"`
- `"HTTP 403"` в сочетании с терминами, связанными с ограничениями
- `"exceeded.*limit"` (шаблон regex)
- `"Please wait"` или `"wait a few minutes"`
- `"abuse detection"` (механизм обнаружения злоупотреблений GitHub)
- `"too many requests"`

### 2. Различия в поведении API

#### Search API (`gh search issues`)

- **Ограничение частоты запросов**: 30 запросов в минуту
- **Вторичные ограничения**: Срабатывают при большом размере страниц (>100) или быстрых запросах
- **Лимит размера страницы**: Практический максимум ~100–200 элементов
- **Поведение**: Более строгие ограничения, более склонен к достижению лимита
- **Сценарий использования**: Поиск в нескольких репозиториях, область организации/пользователя

#### API списка репозиториев (`gh issue list --repo`)

- **Ограничение частоты запросов**: Выше, чем у Search API
- **Вторичные ограничения**: Менее склонен к срабатыванию
- **Лимит размера страницы**: Может успешно обрабатывать 1000+ элементов
- **Поведение**: Более надёжный, лучше для больших наборов данных
- **Сценарий использования**: Один репозиторий, резервная стратегия

### 3. Максимальные размеры страниц

| Тип API        | Рекомендуемый макс. | Проверенный макс. | Примечания                                       |
| -------------- | ------------------- | ----------------- | ------------------------------------------------ |
| Search API     | 100                 | 200               | Значения выше вызывают вторичные ограничения     |
| Repository API | 1000                | 1000+             | Значительно надёжнее при большом размере страниц |
| PR Listing     | 1000                | 1000+             | Аналогично Repository API                        |

### 4. Заголовки ограничения частоты запросов

Ответы GitHub API содержат информацию об ограничениях в заголовках:

```
X-Ratelimit-Limit: 30
X-Ratelimit-Remaining: 18
X-Ratelimit-Reset: 1758160590
X-Ratelimit-Resource: search
X-Ratelimit-Used: 12
```

- Использует `execSync` для команд GitHub CLI
- Реализует пагинацию с улучшенными лимитами (1000)

### Требуемые улучшения

1. **Добавить функцию обнаружения ограничения частоты запросов**

```javascript
function isRateLimitError(error) {
  const errorText = (error.stderr?.toString() || error.stdout?.toString() || error.message || '').toLowerCase();

  const rateLimitPatterns = [/rate limit/i, /secondary rate limit/i, /exceeded.*limit/i, /abuse detection/i, /too many requests/i, /please wait.*before/i, /wait.*(?:few )?minutes?/i, /http 403.*(?:rate|limit|abuse)/i];

  return rateLimitPatterns.some(pattern => pattern.test(errorText));
}
```

2. **Реализовать резервную стратегию**

// В fetchAllIssuesWithPagination

3. **Обновить стратегию размера страниц**

- Search API: использовать максимум 100 элементов на запрос
- Repository API: использовать до 1000 элементов на запрос

### Логика резервного переключения

При достижении ограничений частоты запросов Search API:

1. Обнаружить ошибку ограничения с помощью приведённых выше шаблонов
2. Разобрать исходную команду для извлечения информации о репозитории
3. Преобразовать команду поиска в команду списка репозитория
4. Использовать более высокие лимиты страниц (1000) для Repository API
5. Вернуть результаты в том же формате, что и Search API
