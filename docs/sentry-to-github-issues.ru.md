# Преобразование задач Sentry в задачи GitHub — Исследовательский отчёт (languages: [en](sentry-to-github-issues.md) • [zh](sentry-to-github-issues.zh.md) • [hi](sentry-to-github-issues.hi.md) • ru)

## Обзор

Этот документ рассматривает все доступные варианты автоматического преобразования задач Sentry в задачи GitHub для проекта `link-assistant/hive-mind`. Наш дашборд Sentry доступен по адресу: https://deepassistant.sentry.io/issues

## Текущий статус интеграции

В проекте в настоящее время интегрирован Sentry для отслеживания ошибок:

- **Sentry SDK**: `@sentry/node` (v10.15.0) и `@sentry/profiling-node` (v10.15.0)
- **Реализация**: Комплексная библиотека Sentry по адресу `src/sentry.lib.mjs` с отслеживанием ошибок, breadcrumbs и мониторингом производительности
- **Автоматизация создания задач GitHub отсутствует**

## Доступные варианты

### Вариант 1: Нативная интеграция Sentry с GitHub (через интерфейс)

Sentry предоставляет встроенную интеграцию с GitHub, которую можно настроить через веб-интерфейс Sentry.

#### Возможности:

- **Автоматическое создание задач**: Создание задач GitHub автоматически через правила оповещений
- **Ручное создание задач**: Создание и привязка задач GitHub из интерфейса Sentry
- **Двунаправленная связь**: Привязка задач Sentry к существующим задачам/PR GitHub
- **Интеграция с владельцами кода**: Синхронизация файла CODEOWNERS для автоматических предложений исполнителей
- **Отслеживание коммитов**: Автоматическое разрешение задач Sentry, когда коммиты упоминают `fixes <SENTRY-SHORT-ID>`
- **Комментарии к PR**: Автоматические комментарии к слитым PR, предположительно вызвавшим проблемы

#### Настройка:

1. Перейдите в **Settings > Integrations > GitHub** в Sentry
2. Установите интеграцию GitHub (рекомендуется устанавливать из Sentry, а не из GitHub)
3. Настройте оповещения о задачах для автоматического создания задач GitHub
4. Настройте правила оповещений с действием «Create a new GitHub issue»

#### Ограничения:

- **Требует ручной настройки через интерфейс** для правил оповещений
- **Не управляется программно** — ограниченный доступ API к встроенной интеграции
- **Требует плана Business или Enterprise** для автоматического создания задач
- **Требует плана Team или выше** для ручного управления задачами

#### Влияние на стоимость:

- Может потребоваться обновление плана в зависимости от текущей подписки Sentry

### Вариант 2: Кастомный скрипт с использованием Sentry API + GitHub API

Создайте кастомный скрипт Node.js, который периодически получает задачи Sentry и создаёт соответствующие задачи GitHub.

#### Подход к реализации:

**Шаг 1: Получение задач Sentry**

```javascript
// Using Sentry REST API v0
const response = await fetch('https://sentry.io/api/0/organizations/{org_slug}/issues/?query=is:unresolved', {
  headers: {
    Authorization: 'Bearer <SENTRY_AUTH_TOKEN>',
  },
});
```

**Детали конечной точки:**

- **URL**: `GET /api/0/organizations/{organization_id_or_slug}/issues/`
- **Аутентификация**: Bearer-токен с областью `event:read`
- **Параметры запроса**:
  - `query`: Фильтр задач (по умолчанию: `is:unresolved issue.priority:[high,medium]`)
  - `statsPeriod`: Временной период (`24h`, `7d` и др.)
  - `project`: Фильтр по ID проекта
  - `sort`: Порядок сортировки (`date`, `new`, `freq`, `user`)
  - `limit`: Макс. 100 на запрос

**Шаг 2: Создание задач GitHub**

```javascript
// Using GitHub REST API
const response = await fetch('https://api.github.com/repos/{owner}/{repo}/issues', {
  method: 'POST',
  headers: {
    Authorization: 'Bearer <GITHUB_TOKEN>',
    Accept: 'application/vnd.github+json',
  },
  body: JSON.stringify({
    title: sentryIssue.title,
    body: `**Sentry Issue:** ${sentryIssue.permalink}\n\n${sentryIssue.metadata.value}`,
    labels: ['sentry', 'bug'],
  }),
});
```

**Шаг 3: Отслеживание синхронизированных задач**

- Храните сопоставление между ID задач Sentry и номерами задач GitHub
- Предотвращайте дублирование создания задач
- Варианты хранения:
  - Локальный JSON-файл
  - База данных
  - Метки/метаданные задач GitHub
  - Теги Sentry

#### Варианты планирования:

1. **Cron Job**: Запуск скрипта периодически (например, каждый час)
2. **GitHub Actions**: Использование запланированного рабочего процесса
3. **Таймер systemd**: Для серверных развёртываний
4. **Docker-контейнер**: С планировщиком

#### Преимущества:

- ✅ Полный контроль над форматом и содержимым задач
- ✅ Нет дополнительных зависимостей от сервисов
- ✅ Настраиваемая фильтрация и логика приоритизации
- ✅ Возможность добавления кастомных меток, исполнителей и метаданных
- ✅ Работает с любым планом Sentry
- ✅ Легко интегрируется в существующую кодовую базу

#### Недостатки:

- ❌ Требует обслуживания
- ❌ Не в реальном времени (зависит от интервала опроса)
- ❌ Необходима обработка ограничений частоты запросов обоих API
- ❌ Необходимо реализовать отслеживание состояния во избежание дублирования
- ❌ Требует безопасного хранения токенов

#### Оценка трудозатрат:

- **Начальная разработка**: 4–6 часов
- **Тестирование и уточнение**: 2–3 часа
- **Итого**: 6–9 часов

### Вариант 3: Автоматизация на основе webhook с GitHub Actions

Используйте webhook Sentry для запуска рабочих процессов GitHub Actions, которые создают задачи в режиме реального времени.

#### Архитектура:

```
Sentry Issue Event → Webhook → GitHub Actions Workflow → Create GitHub Issue
```

#### Шаги реализации:

**Шаг 1: Создание внутренней интеграции Sentry**

1. Перейдите в **Settings > Developer Settings > Internal Integrations** в Sentry
2. Создайте новую внутреннюю интеграцию
3. Подпишитесь на события webhook: `issue.created`, `issue.updated`
4. Установите URL webhook для приёмника GitHub Actions webhook

**Шаг 2: Настройка приёмника webhook GitHub Actions**

- Используйте события repository dispatch или прокси webhook
- Варианты:
  - **Прокси-сервис webhook** (например, smee.io для разработки)
  - **Self-hosted приёмник webhook**
  - **Cloud function** (AWS Lambda, Google Cloud Functions)

**Шаг 3: Рабочий процесс GitHub Actions**

```yaml
name: Create GitHub Issue from Sentry
on:
  repository_dispatch:
    types: [sentry-issue]

jobs:
  create-issue:
    runs-on: ubuntu-latest
    steps:
      - name: Create GitHub Issue
        uses: actions/github-script@v7
        with:
          script: |
            const sentryIssue = context.payload.client_payload;
            await github.rest.issues.create({
              owner: context.repo.owner,
              repo: context.repo.repo,
              title: sentryIssue.data.issue.title,
              body: `Sentry Issue: ${sentryIssue.data.issue.web_url}`,
              labels: ['sentry', 'automated']
            });
```

#### Структура полезной нагрузки webhook:

```json
{
  "action": "created",
  "installation": {
    "uuid": "<installation-uuid>"
  },
  "data": {
    "issue": {
      "id": "<issue-id>",
      "title": "Issue title",
      "web_url": "https://sentry.io/...",
      "project": {...},
      "metadata": {...}
    }
  },
  "actor": {
    "type": "application",
    "id": "sentry",
    "name": "Sentry"
  }
}
```

#### Преимущества:

- ✅ Создание задач в реальном времени
- ✅ Управляемый событиями (опрос не требуется)
- ✅ Не требует дополнительной инфраструктуры при использовании cloud functions
- ✅ Масштабируемость
- ✅ Работает с любым планом Sentry

#### Недостатки:

- ❌ Требует настройки конечной точки webhook
- ❌ Более сложная начальная настройка
- ❌ Необходима обработка аутентификации и верификации webhook
- ❌ Потенциальные сбои доставки webhook требуют логики повторных попыток
- ❌ GitHub Actions имеет ограничения использования

#### Оценка трудозатрат:

- **Начальная разработка**: 6–8 часов
- **Тестирование и развёртывание**: 3–4 часа
- **Итого**: 9–12 часов

  - Лёгкая кастомизация с помощью Node.js

**Время настройки**: 15–30 минут

- **Интеграция**: Автоматизация рабочих процессов GitHub + Sentry.io
- **URL**: https://n8n.io/integrations/github/and/sentryio/

## Справочник по API

### Sentry API

**Список задач организации:**

```
GET https://sentry.io/api/0/organizations/{org_slug}/issues/
Authorization: Bearer <token>
```

**Параметры запроса:**

- `query`: Запрос фильтра (например, `is:unresolved issue.priority:high`)
- `statsPeriod`: Временной период (`24h`, `7d`, `14d`)
- `project`: ID проектов для фильтрации
- `sort`: Порядок сортировки (`date`, `new`, `freq`, `user`)
- `limit`: Макс. 100 результатов

**Ответ:**

```json
[
  {
    "id": "issue-id",
    "title": "Error title",
    "permalink": "https://sentry.io/organizations/.../issues/...",
    "project": {
      "name": "Project Name",
      "slug": "project-slug"
    },
    "status": "unresolved",
    "level": "error",
    "count": 42,
    "userCount": 10,
    "firstSeen": "2025-10-01T00:00:00Z",
    "lastSeen": "2025-10-01T12:00:00Z"
  }
]
```

### GitHub API

**Создание задачи:**

```
POST https://api.github.com/repos/{owner}/{repo}/issues
Authorization: Bearer <token>
Accept: application/vnd.github+json
```

**Тело запроса:**

```json
{
  "title": "Issue title",
  "body": "Issue description with Sentry link",
  "labels": ["bug", "sentry"],
  "assignees": ["username"]
}
```

**Ответ:**

```json
{
  "id": 123,
  "number": 456,
  "state": "open",
  "title": "Issue title",
  "html_url": "https://github.com/owner/repo/issues/456"
}
```

## Соображения безопасности

### Токены аутентификации

1. **Токен Sentry Auth**: Создайте с минимальными областями (`event:read`)
2. **Токен GitHub**: Используйте детализированный PAT с разрешением `issues:write`
3. **Хранение**: Используйте переменные среды или безопасное управление секретами
4. **Ротация**: Реализуйте политику ротации токенов

### Безопасность webhook

1. **Верификация подписи**: Проверяйте подписи webhook из Sentry
2. **Только HTTPS**: Всегда используйте HTTPS для конечных точек webhook
3. **Разрешение IP**: При возможности ограничьте источники webhook IP-адресами Sentry
4. **Ограничение частоты запросов**: Реализуйте ограничение частоты на конечных точках webhook

### Конфиденциальность данных

1. **Обработка PII**: Будьте осторожны со стектрейсами, содержащими конфиденциальные данные
2. **Сообщения об ошибках**: Очищайте сообщения об ошибках перед созданием задач GitHub
3. **Управление доступом**: Убедитесь, что репозиторий GitHub имеет соответствующие ограничения доступа
4. **Соответствие требованиям**: Учитывайте требования GDPR/конфиденциальности для данных об ошибках
