# Docker‑изоляция: DinD против DooD (languages: [en](DOCKER-ISOLATION.md) • [zh](DOCKER-ISOLATION.zh.md) • [hi](DOCKER-ISOLATION.hi.md) • ru)

Hive Mind может запускать каждую задачу в её собственном Docker‑контейнере с
помощью `--isolation docker` (об окружающей настройке Docker см.
[DOCKER.md](./DOCKER.ru.md)). На этой странице объясняются **два способа**, которыми
изоляция может общаться с Docker‑демоном — **DinD** и **DooD** — компромисс между
ними и точный рецепт запуска для каждого.

> **Коротко** — на хосте с ограниченным диском предпочитайте **DooD**: бот
> разделяет хостовый Docker‑демон, поэтому изолированные задачи **переиспользуют
> хостовую копию образа с нулевым копированием, нулевым pull и нулевым
> дополнительным диском**. DinD даёт каждому боту свой вложенный демон, но должен
> хранить **вторую полную копию** многогигабайтного образа.
> См. [issue #1962](https://github.com/link-assistant/hive-mind/issues/1962).

## Раннер один и тот же — отличается только демон

`--isolation docker` всегда выдаёт **одну и ту же** простую команду через
start‑command:

```text
$ --isolated docker --image <ref> [--privileged] --shell sh -e … --volume … \
    --detached --session <uuid> -- '<command>'
```

то есть обычный `docker run` против **того демона, с которым общается `docker`
бота**. Режим — это исключительно про то, **какой это демон**:

| Режим                           | Какой демон выполняет задачу                  | Стоимость образа                                                   | Изоляция на задачу                         |
| ------------------------------- | --------------------------------------------- | ------------------------------------------------------------------ | ------------------------------------------ |
| **DinD** (Docker‑in‑Docker)     | **вложенный** демон внутри контейнера бота    | в вложенном хранилище должна лежать **полная вторая копия** образа | контейнер на задачу **и** приватный демон  |
| **DooD** (Docker‑out‑of‑Docker) | **хостовый** демон (разделён через его сокет) | **ноль** — задача переиспользует существующий образ хоста          | контейнер на задачу; **демон разделяется** |

Оба режима дают каждой задаче собственный контейнер (изоляция процессов /
файловой системы / сети). Разница в демоне: DinD вкладывает один (полная копия
образа, больше изоляции); DooD разделяет хостовый (нулевое копирование —
единственный вариант без копирования, когда свободного диска не хватает на вторую
копию образа).

## Как Hive Mind выбирает режим

Hive Mind определяет режим в порядке приоритета:

1. **`HIVE_MIND_DOCKER_ISOLATION_MODE`** — явный `dind` или `dood`. Используйте,
   чтобы быть однозначным.
2. **`DIND_SKIP_DAEMON`** истинно — переключатель DooD у box. Entrypoint DinD
   пропускает запуск вложенного демона, поэтому CLI `docker` нацелен на хостовый
   демон → **DooD**.
3. **`DOCKER_HOST`** указывает на невложенный демон (`tcp://…`, `ssh://…` или
   сокет `unix://`, который **не** является внутриконтейнерным значением по
   умолчанию `/var/run/docker.sock`) → **DooD**.
4. Иначе → **DinD** (историческое значение по умолчанию, так что существующие
   развёртывания не меняются).

С `--verbose` (или `TELEGRAM_BOT_VERBOSE=true`) лог запуска печатает
определённый режим и то, на какой демон нацелен `docker`, поэтому неправильная
конфигурация видна сразу.

## Рецепт DinD (вложенный демон)

Каждая задача выполняется на демоне, вложенном внутрь контейнера бота. Вложенное
хранилище изначально **пусто**, поэтому образ нужно засеять в него (passthrough
хостового образа у box), иначе первая задача скачает полный многогигабайтный
образ. Это подробно описано в [DOCKER.md → Host‑image passthrough](./DOCKER.ru.md#host-image-passthrough-avoid-re-downloading-multi-gb-images):

```bash
docker run -dit --privileged --name hive-mind --restart unless-stopped \
  # ... ваши обычные монтирования учётных данных ...
  -v /var/run/docker.sock:/var/run/host-docker.sock:ro \
  -e DIND_HOST_PASSTHROUGH_IMAGES="konard/hive-mind konard/hive-mind-dind" \
  konard/hive-mind-dind:latest bash -l -c 'bash /home/box/start-bot.sh'
```

DinD стоит диска из‑за дублированного образа, но каждый бот получает полностью
приватный демон. Предпочитайте его, когда изоляция демона важнее диска.

## Рецепт DooD (разделяемый хостовый демон) — рекомендуется, когда диска мало

Бот разделяет **хостовый** демон, монтируя хостовый Docker‑сокет как
`/var/run/docker.sock` и пропуская вложенный демон. Изолированные задачи тогда
выполняются на хостовом демоне, **переиспользуя образ хоста без pull и без
копирования**:

```bash
# GID группы docker на хосте — контейнеру он нужен для чтения смонтированного сокета.
HOST_DOCKER_GID="$(getent group docker | cut -d: -f3)"

docker run -dit --name hive-mind --restart unless-stopped \
  # ... ваши обычные монтирования учётных данных ...
  -v /var/run/docker.sock:/var/run/docker.sock \
  --group-add "${HOST_DOCKER_GID}" \
  -e DIND_SKIP_DAEMON=1 \
  -e HIVE_MIND_DOCKER_ISOLATION_MODE=dood \
  konard/hive-mind-dind:latest bash -l -c 'bash /home/box/start-bot.sh'
```

Ключевые флаги:

- `-v /var/run/docker.sock:/var/run/docker.sock` — `docker` бота теперь общается с
  **хостовым** демоном (а не с вложенным).
- `--group-add <host-docker-gid>` — **обязателен**, чтобы пользователь `box` без
  root мог читать смонтированный сокет; без него `docker` падает с ошибкой прав.
- `-e DIND_SKIP_DAEMON=1` — сообщает entrypoint образа DinD не запускать
  собственный демон (нечего вкладывать).
- `-e HIVE_MIND_DOCKER_ISOLATION_MODE=dood` — делает режим явным, чтобы
  диагностика описывала **хостовый** демон и никогда ложно не предупреждала о
  вложенном демоне или passthrough, которых в DooD нет. (Установка
  `DIND_SKIP_DAEMON` уже выводит DooD; это делает выбор однозначным.)

> **Один образ, оба режима.** `konard/hive-mind-dind` работает в **любом** режиме —
> разница лишь в флагах запуска выше. Отдельный образ для DooD не нужен.

> **Замечание о безопасности.** DooD разделяет хостовый демон, поэтому задачи
> могут добраться до каждого контейнера и образа на хосте. Используйте его на
> хостах, которые вы контролируете, где такая граница доверия приемлема.

## Требование точного тега (оба режима)

`resolveDockerIsolationImageTag()` заставляет каждую задачу запрашивать **точный**
`HIVE_MIND_DOCKER_ISOLATION_IMAGE_TAG` (например, `konard/hive-mind-dind:2.0.13`), а
не плавающий `:latest`. Для переиспользования с нулевым копированием демон должен
держать **именно этот конкретный тег**:

- **DooD** — скачайте точный тег на **хосте** перед запуском задач:
  ```bash
  docker pull konard/hive-mind-dind:2.0.13
  ```
- **DinD** — засейте **вложенный** демон точным тегом (passthrough или скрипт
  предзагрузки в [DOCKER.md](./DOCKER.ru.md#host-image-passthrough-avoid-re-downloading-multi-gb-images)).

Релизные образы запекают `HIVE_MIND_DOCKER_ISOLATION_IMAGE_TAG` из опубликованной
`HIVE_MIND_VERSION`, поэтому родитель, запущенный как `:latest`, всё равно
запускает дочерние контейнеры с того же неизменяемого релизного тега. Зафиксируйте
эту разрешённую версию в своём развёртывании; если у демона есть только `:latest`,
дрейф digest вынудит свежий многогигабайтный pull.

## Проверка переиспользования в DooD (без тихого повторного pull)

Две проверки подтверждают, что бот в DooD и переиспользует образ хоста:

```bash
# 1. docker бота достигает ХОСТОВОГО демона (доступ DooD в порядке).
docker exec hive-mind docker info >/dev/null && echo "DooD docker access OK"

# 2. Точный тег изоляции уже присутствует на этом демоне (переиспользование без копирования).
TAG="$(docker exec hive-mind printenv HIVE_MIND_DOCKER_ISOLATION_IMAGE_TAG || true)"
docker exec hive-mind docker image inspect "konard/hive-mind-dind:${TAG:-latest}" >/dev/null \
  && echo "Образ присутствует на хостовом демоне → ноль копий, нет pull на первой задаче"
```

Стартовый preflight автоматически выполняет эквивалентную проверку и логирует в
режиме DooD:

- ✅ образ присутствует на **хостовом** демоне → задачи переиспользуют его (ноль
  копий / ноль pull);
- ⚠️ образ **отсутствует** на хостовом демоне → скачайте точный тег на хосте и
  зафиксируйте `HIVE_MIND_DOCKER_ISOLATION_IMAGE_TAG` (никогда не упоминает
  вложенный демон или passthrough, которых в DooD нет);
- ⚠️ хостовый демон на драйвере хранилища `vfs` / мало свободного диска — обычные
  предупреждения о переполнении диска, нацеленные на **хостовый** демон.
