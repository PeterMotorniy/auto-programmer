## 痛点

团队和独立开发者都面临一系列反复出现的困扰。Hive Mind 的存在就是为了消除它们。

- **无穷无尽的“小而真实”的待办。** Bug 报告、不稳定的测试、过时的文档、依赖升级、
  小型重构、“适合新手的 issue”——每一项单独看都很便宜，但合在一起永无止境。它们
  挤占了更重要的工作，并且很少被真正完成。
- **为 AI 工具当保姆。** 大多数 AI 编码助手每隔几秒就要请求一次许可，只有在你盯着
  看时才工作，并且一旦关闭编辑器就丢失所有上下文。你变成了 AI 的监工，而不是其
  成果的评审者。
- **不安全的本地执行。** 在你的开发机上给 AI 完全的自由，会让你的令牌、文件和生产
  系统面临风险。沙箱方案虽然安全，却又过于受限，无法安装软件包、浏览文档或配置一个
  真实的环境。
- **上下文切换扼杀心流。** 一次为修小问题而花的两小时打断，一旦算上重新聚焦于真正
  重要工作的代价，远不止两小时。
- **你被绑在电脑前。** 触发、引导和评审 AI 的工作通常需要 SSH 访问、终端和 IDE——
  因此你一离开工位，什么都不会发生。
- **工作无法扩展。** 当十个 issue 同时到来，单个人（或单个 AI 会话）只能逐一处理。
  没有简单的办法按需启动“一个团队”。
- **令牌浪费在管道工作上。** AI 预算被消耗在例行的 git 操作、创建分支和 PR 模板
  上，而不是真正用于解决问题。

共同的主线是：**软件工作中例行的部分，吞噬了本应投入创造性部分的时间、注意力和
金钱。**

## 愿景

> **Hive Mind 是一个总指挥 AI，它编排一群 AI 工作者——并在需要时编排集体的人类
> 智慧——来清理软件中的例行工作，从而把人的注意力留给那些真正需要它的决策。**

我们相信正确的分工是：

- **AI 做例行工作**——阅读代码库、编写改动、创建分支和拉取请求、反复迭代直到检查
  通过。
- **人类做决策**——定义问题、在需求不明确时回答问题，并决定合并什么。
- **自动化做管道工作**——git、fork、PR 创建、会话恢复和编排都在代码中完成，因此
  AI 的令牌和人的注意力都能聚焦在创造性的核心上。

我们正在迈向的终态：你从任何设备描述一个问题，去睡觉，醒来时迎接一个可供评审的拉取
请求草稿——或者，如果问题描述不充分，迎接一个深思熟虑的提问。把这一切从一个 issue
扩展到整个待办，从一个工作者扩展到一个协同的蜂群。

## 指导原则

这些原则解释了 Hive Mind 刻意做出的取舍。

1. **自主胜过当保姆。** 工作者以完全自主的模式运行。人的接触点是开始（issue）和
   结束（评审），而不是中间的每一条命令。
2. **隔离胜过本地风险。** 工作者运行在一次性的虚拟机或 Docker 容器中，而不是
   （推荐做法）你的开发机上。损坏的环境是被还原，而不是被修复。
3. **在合并关口由人把关。** AI 产出 **草稿** 拉取请求。没有人的决定，任何东西都
   不会进入你的主分支。分支保护得到尊重。
4. **令牌与注意力的效率。** 例行步骤是代码，而非提示词。昂贵的、创造性的预算只
   花在需要创造力的地方。
5. **默认通才。** 几乎任何能用仓库中文件完成的事都在范围之内——代码、文档、配置、
   测试、数据——而不仅仅是编程。
6. **在用户所在之处与他们相遇。** GitHub 和 Telegram 在每一部手机上都有，因此
   管理蜂群不应当需要一台笔记本。
7. **开放且可移植。** Unlicense 许可（公有领域）。没有供应商锁定；在我们的基础
   设施或你自己的基础设施上运行。

## 面向谁

| 角色                  | 他们的痛点                            | Hive Mind 提供的                                        |
| --------------------- | ------------------------------------- | ------------------------------------------------------- |
| **独立开发者**        | 小 issue 的待办偷走了真正工作的时间   | 一位不知疲倦的队友，在你专注或睡觉时起草修复            |
| **团队负责人 / 经理** | 例行 issue 堆积；评审在人这里形成瓶颈 | 一群蜂群，一夜之间清空待办，只留下评审                  |
| **开源维护者**        | issue 太多，贡献者太少                | 自动 fork 的工作者，把 issue 变成可供评审的 PR          |
| **贡献者**            | 想帮忙，但没有写权限或没有搭建时间    | fork 并解决的流程，无需仓库权限或本地工具链             |
| **非程序员 / PM**     | 能描述问题，但无法实现                | issue 进、PR 出的流程，完全在 GitHub 或 Telegram 中驱动 |
| **组织 / 平台团队**   | 需要可扩展、可观测、可自托管的自动化  | Kubernetes/Helm 部署、按账户隔离、Sentry 跟踪           |

## 我们今天已支持的用户旅程

每个旅程都列出 **角色**、**痛点**、**步骤** 和 **结果**。所示命令均为真实命令；
完整语法见 [README](../README.zh.md)。

### 旅程 1 — 清理待办中的单个 issue

- **角色：** 独立开发者
- **痛点：** 一个已知的 bug 在待办里躺了好几周，因为它无聊，而非困难。
- **步骤：**
  1. 编写（或选取）一个描述该 bug 的 GitHub issue。
  2. 运行 `solve https://github.com/owner/repo/issues/123 --model opus`。
  3. 工作者启动一个隔离环境，阅读代码库，复现 bug，编写修复和测试，并开启一个
     **草稿** 拉取请求。
  4. 评审该 PR。合并它，或评论以请求修改。
- **结果：** 原本要花两小时上下文切换的事，变成对一份完成的草稿做十分钟的评审。

### 旅程 2 — 通过评论迭代拉取请求（继续模式）

- **角色：** 评审者 / 独立开发者
- **痛点：** 第一版草稿对了 80%，但漏掉了一个边界情况。
- **步骤：**
  1. 留下一条普通的 PR 评审评论，描述需要修改的内容。
  2. 运行 `solve https://github.com/owner/repo/pull/456 --verbose`（或让编排器
     自动拾取该评论）。
  3. 工作者阅读反馈，更新解决方案，并推送到同一分支。
  4. 反复进行直到满意，然后合并——或关闭 PR 以立即停止工作。
- **结果：** 改进通过正常的 GitHub 评审流程完成，无需学习新工具。完整的反馈循环
  见 [flow.zh.md](./flow.zh.md)。

### 旅程 3 — 为你没有写权限的仓库做贡献

- **角色：** 开源贡献者
- **痛点：** 你想修复别人项目中的一个 issue，但既无写权限，也没有为其技术栈搭建
  本地环境。
- **步骤：**
  1. 运行 `solve https://github.com/owner/repo/issues/123 --fork --model opus`
     （缺少写权限时也会自动创建 fork）。
  2. 工作者 fork 仓库，在 fork 的分支上解决 issue，并开启一个跨仓库的拉取请求。
  3. 上游维护者评审并合并。
- **结果：** 无需权限、无需克隆、也无需在本地安装项目工具链即可做出贡献。

### 旅程 4 — 一夜清空整个待办（编排）

- **角色：** 团队负责人 / 维护者
- **痛点：** 几十个 issue 都开着；逐一处理要花上数周。
- **步骤：**
  1. 把编排器对准一个仓库、用户或组织：
     `hive https://github.com/owner/repo --monitor-tag "bug" --concurrency 4`。
  2. 编排器持续发现匹配的 issue，并把每一个分配给一个并行的工作者（可用选项如
     `--all-issues`、`--skip-issues-with-prs`、`--max-issues`、`--interval`）。
  3. 每个工作者独立地为自己的 issue 起草一个 PR。
  4. 早上，一队草稿 PR 正等待评审。
- **结果：** 一群蜂群感觉就像一支开发团队，而成本只是一份订阅。
  _“代码在你睡觉时被写出来。”_

### 旅程 5 — 用手机管理蜂群（Telegram）

- **角色：** 任何离开工位的人
- **痛点：** 灵感（或事故）在你途中袭来，而身边没有笔记本。
- **步骤：**
  1. 向 Telegram 机器人发送 `/solve <issue-url>` 或 `/hive <repo-url>`。
  2. 用 `/limits` 查看进度；在群聊中与团队协作。
  3. 从 GitHub 手机应用评审并合并最终的 PR。
- **结果：** 真正的移动开发——创建 issue、触发求解器、评审 PR，全部在一部手机上
  完成。无需 PC、IDE 或“vibe coding”应用。

### 旅程 6 — 自动化非代码工作（通才 AI）

- **角色：** 维护者 / 技术写作者 / PM
- **痛点：** 文档漂移、配置需要更新、数据文件需要重新生成——这些都不是“编程”，
  但都耗费时间。（本文档本身就是通过这一流程产出的。）
- **步骤：**
  1. 开一个描述文档、配置或数据改动的 issue。
  2. 像对待任何其他 issue 一样对它运行 `solve`。
  3. 工作者编辑相关文件并开启一个草稿 PR。
- **结果：** 几乎任何可以表达为仓库文件的东西都能被自动化，而不仅仅是源代码。

### 旅程 7 — 并行运行两个预算与工具（多工具）

- **角色：** 高级用户 / 小团队
- **痛点：** 单个订阅的用量上限或单个模型的风格并不适合每一个任务。
- **步骤：**
  1. 同时配置 Claude MAX（`--tool claude`，默认）和 ChatGPT Pro
     （`--tool codex`）。
  2. 把创造性任务路由给 Claude，把确定性重构路由给 Codex（`/codex` 或
     `/solve --tool codex`），或让并行的工作者混用工具。
- **结果：** 两个独立的、几乎“无限”的预算，以及按工具/模型的并发，因此吞吐量
  不再受限于单个账户。

### 旅程 8 — 安全地大规模自托管

- **角色：** 组织 / 平台团队
- **痛点：** 你需要一种隔离、可观测、且运行在你自己基础设施上的自动化。
- **步骤：**
  1. 通过 Docker、Ubuntu 一条命令安装，或用于生产扩展的 Helm/Kubernetes 部署。
  2. 按容器隔离账户；轮换令牌；接入 Sentry 进行错误跟踪。
  3. 针对你的仓库运行编排，并发度按你的硬件调优。
- **结果：** 一次性、可还原、可监控的 AI 工作者，运行在你的策略所要求的地方。
  见 [DOCKER.zh.md](./DOCKER.zh.md) 和 [HELM.zh.md](./HELM.zh.md)。

### 旅程 9 — 挺过用量上限与中断

- **角色：** 任何执行长任务的用户
- **痛点：** 模型在执行到一半时撞上用量上限，而幼稚的工具会直接失败。
- **步骤：**
  1. 撞上上限的工作者会记录一条恢复命令，并能在上限重置时自动继续。
  2. 必要时显式恢复：`solve <issue-url> --resume <session-id>`。
  3. 通过 `--attach-logs` 和 `--auto-attach-solution-summary` 把日志和解决方案
     摘要附到 PR 上，以保证透明度。
- **结果：** 长任务跨越中断而完成，而不是丢失。

## 我们希望支持的用户旅程（未来）

这些旅程描述了项目的 **方向**。它们并非都已实现；列出它们是为了让文档既涵盖我们
今天支持的，也涵盖我们未来可能支持的。

### 未来旅程 A — 用于任务拆解的架构师 AI

- **角色：** 面对一个庞大、多步骤特性的开发者
- **今天的局限：** Hive Mind 擅长小而界定清晰的任务；大型特性必须先由人拆解。
- **愿景：** 一个“架构师”AI 把大 issue 拆解为一张小子 issue 的图，把它们派发给
  蜂群，并执行最终的集成评审——把一个大请求变成许多并行的草稿。

### 未来旅程 B — 把集体人类智慧纳入循环

- **角色：** 撞上真正知识空白的工作者
- **愿景：** 当需求模糊或需要领域专长时，系统会把一个精确的问题路由给合适的人
  （或一群人）以获取需求、专长或反馈——并在得到答复后自动继续。这个“蜂群”同时
  结合了 AI 工作者 _和_ 人。

### 未来旅程 C — 自主的质量与自我评审循环

- **角色：** 希望一次通过质量更高的维护者
- **愿景：** 专门的评审者代理在人打开 PR 之前，就批判并加固每一份草稿——对抗式地
  验证修复、扩展测试覆盖、标记回归。

### 未来旅程 D — 对仓库的持续看护

- **角色：** 长期存续的项目
- **愿景：** 一个常驻的蜂群主动监视仓库的依赖更新、失败或不稳定的测试、文档漂移
  和安全公告，并开启草稿 PR 来处理它们，而无需等人去提 issue。

### 未来旅程 E — 跨仓库与组织级的行动

- **角色：** 在各处推行一项变更的平台团队
- **愿景：** 只描述一次变更（例如“把每个服务从已弃用的 API 迁走”），就让蜂群在
  众多仓库间铺开，在整个组织范围内协同地创建一致的草稿 PR。

> 有这里没有列出的、你需要的旅程吗？这个空白本身就是一个绝佳的 issue——Hive Mind
> 正是为通过这一循环成长而生的。
