# Case Study — Issue #2043: Future-proof Codex model aliases and OpenAI prefixes

- **Issue:** [link-assistant/hive-mind#2043](https://github.com/link-assistant/hive-mind/issues/2043)
- **Pull request:** [#2044](https://github.com/link-assistant/hive-mind/pull/2044)
- **Investigated:** 2026-07-11
- **Affected command:** `solve --tool codex --model <model>` and the Telegram `/codex` alias

## Executive summary

Hive Mind already listed `gpt-5.6-sol`, `gpt-5.6-terra`, and `gpt-5.6-luna`, but
rejected the intended short forms `sol`, `terra`, and `luna`. It also accepted
only a hand-maintained subset of `openai.` model names and accepted no
`openai/` forms for Codex. The issue screenshot captures all three short names
failing validation before Codex starts.

The root cause was duplicated, manually enumerated model data. Mapping,
validation, compatibility, and error-message lists were based on static object
keys. Each new OpenAI model therefore needed every prefix spelling to be added
individually, while generation-relative aliases had no representation at all.

The solution derives one Codex model-variant map from the canonical catalog:

1. `sol`, `terra`, and `luna` resolve to the newest generation for which all
   three tiers exist.
2. Both `openai/` and `openai.` forms are generated for every known Codex/OpenAI
   model and for the three short aliases.
3. Mapping, validation, compatibility checks, and valid-model output consume the
   same generated map.

## Reproduction

Before the fix, each of these was rejected as an unrecognized Codex model:

```text
/codex https://github.com/owner/repo/pull/1 --model sol
/codex https://github.com/owner/repo/pull/1 --model terra
/codex https://github.com/owner/repo/pull/1 --model luna
```

The screenshot's available-model list exposed the inconsistency: full
`gpt-5.6-*` IDs and a few `openai.gpt-*` IDs appeared, while the requested
aliases did not.

The minimal automated reproduction is
`tests/test-issue-2043-codex-model-aliases.mjs`. Before implementation it failed
because no generation-aware alias resolver existed. It now checks bare and
prefixed aliases, both prefixes for every known model, rejection of unknown
prefixed names, and a synthetic future `gpt-5.7` catalog.

## Design details

### Complete-family promotion

Aliases advance only when a catalog contains `sol`, `terra`, **and** `luna` for
the same generation. If a future rollout exposes only `gpt-5.7-sol`, all three
aliases stay on GPT-5.6. This avoids an incoherent alias family during a staged
rollout.

Generation comparison uses `localeCompare(..., { numeric: true })`; it correctly
orders dotted numeric generations without adding a versioning dependency. The
model IDs are not full semantic versions, so a general semver library would add
weight without improving correctness.

### Prefix preservation

Prefix normalization preserves the spelling the caller selected:

| Input           | Codex model argument |
| --------------- | -------------------- |
| `sol`           | `gpt-5.6-sol`        |
| `openai/sol`    | `openai/gpt-5.6-sol` |
| `openai.sol`    | `openai.gpt-5.6-sol` |
| `openai/gpt-4o` | `openai/gpt-4o`      |
| `openai.gpt-4o` | `openai.gpt-4o`      |

This implements the repository issue literally while keeping bare names
available for ChatGPT-authenticated Codex installations. An upstream Codex
report documents that provider-prefixed model names can be rejected with
ChatGPT authentication, so Hive Mind does not rewrite a user's explicit prefix
to another form.

### One generated source for all consumers

The pre-fix structure had `codexModels`, `CODEX_MODELS`, and several switch
branches independently using model keys. The generated variant map is now used
by alias resolution, validation, tool compatibility, and valid-model listing.
This prevents a model from resolving in one path but failing in another.

## Options considered

### 1. Add 3 aliases and enumerate every prefixed spelling

Rejected. It fixes today's examples but repeats the defect for the next model
and does not satisfy “future generations” or “all OpenAI models.”

### 2. Accept every string beginning with `openai/` or `openai.`

Rejected. It would turn typos such as `openai/not-a-real-model` into late Codex
runtime failures and weaken Hive Mind's early validation.

### 3. Strip prefixes before invoking Codex

Rejected as the universal behavior. It would discard an explicitly selected
gateway/provider namespace. Bare models remain available where the upstream
CLI/auth mode requires them.

### 4. Generate validated variants from the canonical catalog (selected)

This supports all known models, rejects unknown ones early, preserves the
caller's namespace, and automatically follows complete future tier families.

### Existing components/libraries

- The repository's unified `src/models/index.mjs` remains the correct component;
  centralizing the invariant there keeps every CLI entry point consistent.
- A semver package was considered for generation ordering but is unnecessary:
  `5.6` is a dotted numeric catalog generation rather than a release version,
  and JavaScript numeric collation handles the required ordering.
- Codex itself has a model catalog, but Hive Mind performs validation before
  starting Codex and needs aliases that Codex does not define. Runtime catalog
  discovery is already used for default fallback selection, not stable CLI
  alias semantics.
