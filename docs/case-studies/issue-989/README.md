# Case Study: Issue #989 - Perlbrew Bashrc Unbound Variable Error

## Executive Summary

**Issue:** During Docker image build, the error `/home/hive/perl5/perlbrew/etc/bashrc: line 71: $1: unbound variable` appears during the Perl version check.

**Root Cause:** The perlbrew bashrc file uses positional parameter `$1` without protection against `set -u` (nounset mode), causing an unbound variable error when the `__perlbrew_reinit()` function is called without arguments.

**Impact:** Warning message appears in build logs, potentially confusing users and cluttering CI output.

**Solution:** Upgrade perlbrew to version 1.02+ which includes the fix for `set -u` compatibility, or patch the generated bashrc after installation.

## Error Details

```
#9 696.3 [✓] Perl: This is perl 5, version 42, subversion 0 (v5.42.0) built for x86_64-linux
#9 696.3 /home/hive/perl5/perlbrew/etc/bashrc: line 71: $1: unbound variable
#9 696.3 [✓] Perlbrew:
```

## Timeline of Events

### 1. Issue Discovery (2025-12-21)

- The error was first reported in CI logs during Docker image builds
- Similar issue was tracked in Issue #954

### 2. Previous Fix Attempts

- Issue #954 documented the same error and proposed solutions:
  - Wrapping perlbrew source command with interactive shell check (`$PS1`)
  - Using `set +u` / `set -u` protection when sourcing

### 3. Root Cause Analysis

The error occurs in the `__perlbrew_reinit()` function in perlbrew's generated bashrc file:

```bash
__perlbrew_reinit() {
    if [[ ! -d "$PERLBREW_HOME" ]]; then
        mkdir -p "$PERLBREW_HOME"
    fi

    [ -f "$PERLBREW_HOME/init" ] && rm "$PERLBREW_HOME/init"
    echo '# DO NOT EDIT THIS FILE' > "$PERLBREW_HOME/init"
    command perlbrew env $1 | \grep PERLBREW_ >> "$PERLBREW_HOME/init"  # Line 71: $1 is unbound
    . "$PERLBREW_HOME/init"
    __perlbrew_set_path
}
```

The `$1` on line 71 is not protected with `${1:-}` syntax, causing the unbound variable error when:

1. The script runs with `set -u` (nounset) enabled
2. The function is called without any arguments

## Research Findings

### Perlbrew Version History

According to [metacpan.org](https://metacpan.org/dist/App-perlbrew), perlbrew version 1.02 (released 2025-07-18) includes:

> "The bashrc now works better with `set -u` ('use strict' in bash) is in place. PR #850."

### PR #850 Fix Details

[GitHub PR #850](https://github.com/gugod/App-perlbrew/pull/850) addressed two unbound variable crashes:

1. `$PERLBREW_LIB` on line 45
2. `$outsep` variable in `__perlbrew_purify` function on line 24

The fix wrapped both problematic variable expansions using `${var:-}` syntax.

**However**, the `$1` in `__perlbrew_reinit()` may still be unprotected in some versions, or there's additional code that references `$1`.

## Proposed Solutions

### Solution 1: Upgrade Perlbrew (Recommended)

Upgrade to perlbrew version 1.02+ which includes the `set -u` compatibility fix:

```bash
# After perlbrew installation
perlbrew self-upgrade
```

**Pros:**

- Uses upstream fix
- Future-proof
- Simple to implement

**Cons:**

- Requires network access during build
- May not work if the self-upgrade command itself triggers the error

### Solution 2: Patch Bashrc After Installation

After perlbrew installation, patch the generated bashrc to use `${1:-}`:

```bash
# Patch the perlbrew bashrc to fix unbound variable issue
sed -i 's/\$1/${1:-}/g' "$PERLBREW_ROOT/etc/bashrc"
```

**Pros:**

- Direct fix
- Works regardless of perlbrew version
- No network dependency

**Cons:**

- Modifies third-party files
- May need adjustment if perlbrew format changes

### Solution 3: Wrap Source with set +u Protection (Current Implementation)

The current implementation already uses this approach in `ubuntu-24-server-install.sh`:

```bash
# Temporarily disable unset variable check to avoid perlbrew bashrc errors
set +u
source "$PERLBREW_ROOT/etc/bashrc"
set -u
```

**Why This Isn't Enough:**

This protection only works when sourcing directly in the script. The error occurs during `perl --version` check at line 1367, which may trigger a subshell that sources .bashrc.

### Solution 4: Add CI Check for Unbound Variable Errors (Monitoring)

Add a CI check to detect and fail on unbound variable errors in Docker build logs:

```yaml
if grep -E 'unbound variable' build-output.log; then
  echo "ERROR: Unbound variable error detected in build"
  exit 1
fi
```

**Pros:**

- Enforces that the error is fixed
- Prevents future regressions

**Cons:**

- Doesn't fix the root cause
- May catch legitimate warnings

## Recommended Implementation

1. **Add perlbrew self-upgrade after installation** (Solution 1)
2. **Add bashrc patch as fallback** (Solution 2)
3. **Add CI check for unbound variable errors** (Solution 4)

## Code Changes

### Location: `scripts/ubuntu-24-server-install.sh`

After perlbrew installation, add:

```bash
# Upgrade perlbrew to latest version for set -u compatibility fix
log_info "Upgrading perlbrew to latest version..."
perlbrew self-upgrade 2>/dev/null || log_note "perlbrew self-upgrade skipped"

# Patch bashrc as fallback to fix any remaining unbound variable issues
if [ -f "$PERLBREW_ROOT/etc/bashrc" ]; then
    sed -i 's/\$1/${1:-}/g' "$PERLBREW_ROOT/etc/bashrc" 2>/dev/null || true
    sed -i 's/\$PERLBREW_LIB/${PERLBREW_LIB:-}/g' "$PERLBREW_ROOT/etc/bashrc" 2>/dev/null || true
fi
```

### Location: `.github/workflows/release.yml`

Add CI check in the `docker-pr-check` job:

```yaml
# Check for unbound variable errors
if grep -E 'unbound variable' build-output.log; then
  echo "ERROR: Unbound variable error detected in Docker build"
  cat build-output.log | grep -E 'unbound variable' || true
  exit 1
fi
```
